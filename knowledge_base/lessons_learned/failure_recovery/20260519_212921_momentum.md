## Applied Lessons
Now I have a clear picture. The "unsupported characters" error in WorldQuant Brain typically stems from the `/` (division) operator and the `-` (subtraction) used as `1 - ts_corr(...)`. The WQB parser is picky about inline arithmetic with certain operator combinations.

**Remediation notes:**

1. **Division (`/`) inside `rank()`**: The parser rejects bare division inside nested function calls. Fix: pre-compute the ratio using `ts_delta` and `ts_mean` patterns, or restructure so division is the outermost operation.

2. **Subtraction `1 - ts_corr(...)`**: The literal `1 -` prefix is an unsupported character pattern. Fix: replace with `rank(-ts_corr(...))` and add back the neutral level outside, or use `ts_mean` offsets.

3. **Operator count**: The original has 3 divisions, 1 subtraction, and mixed nesting depth. Reducing operator density improves parse success.

4. **Turnover adjustment**: The `ts_delta(volume,1)/ts_mean(volume,20)` term is a daily volume spike signal â€” high turnover noise. Widening to `ts_delta(volume,5)/ts_mean(volume,20)` smooths it and reduces churn.

5. **Correlation adjustment**: `ts_corr(volume, ts_delta(returns,3), 10)` uses a 3-day return delta which is noisy. Using raw `returns` with a longer lookback (20) stabilizes the correlation estimate and reduces drawdown sensitivity to regime shifts.

6. **Drawdown guard**: The `zscore` wrapper on the momentum term can produce extreme tails. Clipping via `rank(zscore(...))` is fine, but the momentum Sharpe ratio `ts_mean/ts_std_dev` is already normalized â€” dropping the outer `zscore` reduces tail risk.
